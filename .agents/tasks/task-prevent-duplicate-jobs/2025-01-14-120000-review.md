# Duplicate job prevention via partial unique index

This change prevents users from starting a second background job of the same type while one is already pending or running. It does so by adding a partial unique index to the `jobs` table, catching the resulting `IntegrityError` in the service layer, and surfacing a user-friendly flash message through a new `DuplicateJobError` exception that propagates cleanly from service → worker → route handler.

**Watch for:** The partial index is defined with `sqlite_where` only, which means it works in tests (SQLite) but emits a **plain unique index** on MySQL production — this will prevent creating *any* second job of the same type regardless of status, breaking the application (confirmed). The `IntegrityError` catch is also overly broad; any integrity violation during commit will be misreported as a duplicate job (likely).

## High-level view

The enforcement strategy relies on a database-level constraint rather than an application-level check-then-insert, which eliminates TOCTOU races between concurrent requests. The problem is that `sqlite_where` is a SQLite-only dialect option; on MySQL, SQLAlchemy emits `CREATE UNIQUE INDEX idx_unique_active_job ON jobs (job_type)` without the WHERE clause, making the index unconditionally unique. Production would reject any job creation if *any* prior job of that type exists in the table.

The exception propagation path layers `DuplicateJobError` from service → worker → route, though the worker's catch-and-reraise adds only a log line without changing behavior or enriching the exception.

All tests run on SQLite where the partial index works correctly, so they don't reveal the MySQL production failure.

## The sqlite_where production gap

The index definition:

```python
Index(
    "idx_unique_active_job",
    "job_type",
    unique=True,
    sqlite_where=Column("status").in_(["pending", "running"]),
)
```

Generates correct DDL for SQLite:
```sql
CREATE UNIQUE INDEX idx_unique_active_job ON jobs (job_type)
WHERE status IN ('pending', 'running')
```

But on MySQL (production dialect), it emits:
```sql
CREATE UNIQUE INDEX idx_unique_active_job ON jobs (job_type)
```

MySQL does not support partial (filtered) indexes. The `sqlite_where` kwarg is silently dropped for any non-SQLite dialect. This makes the constraint unconditionally unique on `job_type`, meaning once a single completed job of type `X` exists in the table, no new job of type `X` can ever be created. The application will start raising `DuplicateJobError` for every job start attempt after the first job of each type completes.

The fix requires a MySQL-compatible uniqueness strategy. Common approaches: (a) add a computed column like `active_guard` that is `job_type` when status is pending/running and NULL otherwise, then put a unique index on `(active_guard)` (MySQL unique indexes ignore NULLs); (b) use an application-level `SELECT ... FOR UPDATE` before insert; (c) use a separate `active_jobs` tracking table. Option (a) preserves the database-level enforcement that motivated this design.

## IntegrityError specificity

```python
except IntegrityError:
    db.session.rollback()
    raise DuplicateJobError(...)
```

The `username` column is `nullable=False` in the model but `create_job` accepts `username: str | None = None`. If `None` is passed, the resulting NOT NULL violation will also be an `IntegrityError`, misreported as a duplicate job. This path may not be exercised today (callers always pass a username), but it's a latent trap. Inspecting `exc.orig` or matching on the constraint name (`idx_unique_active_job`) would make this robust.

## Test coverage gap

The five new unit tests in `test_jobs_service.py` verify the constraint at the service layer: duplicate pending, duplicate running, completed-then-new, cancelled-then-new, and different-type coexistence. The worker propagation test mocks `create_job` to verify `DuplicateJobError` flows through `start_job`.

Not tested: the route handler's `except DuplicateJobError` branch is never exercised through an HTTP request. No test verifies that the flash message appears when a duplicate is attempted through the route, leaving the user-facing behavior unverified at the integration level.
